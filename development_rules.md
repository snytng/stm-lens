# 開発再開時の手順（新しいセッションを開始する場合）

セッションがリセットされた場合や、長期間間が空いた場合は、以下の手順でコンテキストを再読み込みさせてください。

1.  **ドキュメントの読み込み**: 以下の3つの主要ドキュメントを順番に貼り付け、プロジェクトの全体像と現状を伝えます。
    *   `customer_value.md` （顧客価値）
    *   `requirements.md` （要件定義書）
    *   `specification.md` （仕様書）
    *   `design.md` （設計書）
    *   `implementation_log.md` （実装ログ）
    *   `test_cases.md` （テスト仕様）
2.  **準備完了の確認**: 私がドキュメントを読み込み、次のステップの準備ができたことを確認します。
3.  **ソースコードの提供**: `implementation_log.md`に記載されている「次のステップ」の実装に着手します。
    *   もし、あなたが既に次のステップのコードを書き始めている場合は、そのファイルを貼り付けてください。
    *   まだ手をつけていない場合は、その旨を伝えてください。私の方で最初の実装案を提示します。
4.  **astah* API を確認する**: このプロジェクトでは以下のastah* APIを利用するため
    https://members.change-vision.com/javadoc/astah-api/11_0_0/api/en/doc/javadoc/index.html

# 開発作業の定義
1. プロジェクトの変更作業は、機能追加、不具合修正、ドキュメント修正のいずれかを決めるものとします。
2. 決めた作業の範囲でイテレーション開発を行います。
3. 開発の単位をアクションと呼び、ドキュメントの更新、テストコードの作成、コードの更新、テストの実行を一通り行います。
4. 開発の記録は`implementation_log.md`に記録します。

# ドキュメント更新・開発作業のルール

## 1. 作業着手前
1.  次の作業アクション候補を検討する。私だけが決めるのではなく、geminiさんと一緒に考えます。
2.  私が次の作業アクションを決める。
3.  geminiさんは、決めた内容を`implementation_log.md` の「次のステップ」に通し番号を附番して追記する。もし今までの作業アクションに番号がなければ、順番に番号をつける。
4.  **重要**: 一つの作業アクションが完了しても、私の明示的な指示があるまでは、勝手に次の作業アクションに関する作業を開始してはいけません。

## 2. 要件・仕様・設計の確認
1.  追加した作業アクションが、プロジェクト全体の価値・要件・仕様・設計 (`customer_value.md`, `requirements.md`, `specification.md`, `design.md`) と照らし合わせて妥当か確認する。
2.  もし、既存の要件・仕様・設計と矛盾する、または大きな影響を与える可能性がある場合は、実装に着手する前に開発者に質問し、方向性を確認する。

## 3. ドキュメントの先行更新
1.  作業アクションが妥当である場合、対応する価値・要件・仕様・設計が `customer_value.md`, `requirements.md`, `specification.md`, `design.md` に存在するか確認する。
2.  不足している、または修正が必要な要件・仕様・設計があれば、実装を開始する前に、`customer_value.md`, `requirements.md`, `specification.md`, `design.md` を更新する。

## 4. テスト仕様作成
1.  gemniさんは作業アクションの完了を確認するためのテストを検討する。
2.  私は、検討したテストケースについて作業時間が大きくかかる場合には私に方針を確認する。
3.  geminiさんは、テスト内容が決まったら、以下のテスト資産を作成・更新する。
    *   ロジックの単体テストコード (JUnit): `src/test/java` フォルダ以下に配置する
    *   テストモデル生成スクリプト: `test/scripts/`フォルダ以下に`create_test_model_no##.js`という命名規則で配置する
    *   **生成されるモデル名**: スクリプト内で生成するモデル（図）の名称は `TestModel_No##` とする。
    *   **スクリプトの制限事項**: APIの制約によりスクリプトで完全なモデルを生成できない場合（例: Regionが作れない）、スクリプト内にコメントで制限事項を明記し、実行時にユーザーへ手動修正を促すメッセージを出力するようにする。
    *   テスト手順書: `test/test_cases.md`に記載する
4.  私は、テスト内容を確認し問題があれば1.に戻る。問題なければ次に進む。

## 5. 実装
1.  geminiさんは、作業アクションとドキュメントに問題がないことを確認する。問題があれば私に確認して完了を確認する。
2.  geminiさんは、ソースコードの修正（実装）を開始する。
3.  geminiさんは、私に対して、変更したソースコードに対して、何を変更したか、私が動作確認すべきこと、これで作業アクションが完了するかどうか、完了しない場合は残りの作業の見通し（何をする予定か）を説明する。

## 6. テスト実施
1.  私がビルドおよび自動テスト（Maven test）を実施する。その結果をgeminiさんに伝える。
2.  ビルド・テストが問題があれば1.に戻る。
3.  もし作業アクションのすべて終わったら私がgeminiさんへ明確に終了したことを伝える。
4.  もし作業が終わっていなければ、やり直しをするために、geminiさんは上記手順のやり直し箇所を私に説明する。

## 7. 完了処理
1.  私がgeminiさんへ「実装・ビルド・テストが完了し、作業アクションが終了した」ことを伝える。
2.  geminiさんは、その作業アクションを`implementation_log.md`の「完了」セクションに移動すると同時に、利用者向けドキュメント（`README.md`, `usage.md`, `CHANGELOG.md`など）の更新案（diff）を提示する。
3.  私がドキュメントの変更内容を確認・適用し、すべての変更をgitにコミットする。
4.  私が「作業アクション完了」を宣言し、一つのイテレーションを終了する。

## 8. その他の作業
1.  開発がある程度進んだら以下を実施する。
    1.  ドキュメントの全体的な整合性を再度レビューしてください。
    2.  ソースコードをリファクタリングする。可読性をよくする。重複をなくすことのみ。
2.  私がこのドキュメントで宣言している手順を逸脱したことをgeminiさんに要求したときには、geminiさんはこのドキュメントを引用して私に作業指示が矛盾していることを伝えてよいというか、むしろ歓迎します。

## 9. 開発ノウハウ・注意点 (Tips & Caveats)
*   **astah* API**:
    *   `IRegion` インターフェースは存在しない。領域は `IState` として扱われる。
    *   スクリプトで領域を作成する場合は `StateMachineModelEditor` を使用する (`BasicModelEditor` や `DiagramEditor` ではない)。
    *   スクリプト (`.js`) でモデルを生成・変更する場合は、必ず `transactionManager.beginTransaction()` と `transactionManager.endTransaction()` で処理を囲むこと。
    *   `IVertex.getContainer()` などで階層を辿る際、トップレベルの要素では戻り値が `null` になることを常に考慮し、`null` チェックを行うこと。
*   **ファイル構成**:
    *   単体テストコード (`.java`) は必ず `src/test/java` ディレクトリ配下に配置する。
    *   テスト用スクリプト (`.js`) は必ず `test/scripts` ディレクトリ配下に配置する。`src/` いかには置かない。
*   *   テスト仕様書 (`.md`) は必ず `test/` ディレクトリ配下に配置する。`src/` には置かない。
*   **シミュレーションロジック**:
    *   平行状態からの遷移では、LCA (Least Common Ancestor) を特定し、遷移元からLCAまでのパス上にある状態とその子孫のみを終了させる（他のリージョンの状態を巻き込まないようにする）。
    *   複合状態退出時のハイライトは、退出した親状態だけでなく、その内部で直前までアクティブだった子状態も含めて行うと分かりやすい。
*   **テスト・品質保証**:
    *   異常系（フェイルセーフ）のテストを検討する際は、ソースコードを書き換える手法よりも、**「不正な構造のモデル（不正データ）を入力して異常状態を誘発する」** アプローチを優先して検討する。
*   **ドキュメント更新**:
    *   ユーザーの手により既にドキュメントが更新されている場合があるため、diffを提示する際はコンフリクト（内容の重複適用）にならないよう注意する。

## 10. ファイル作成・更新時のパス指定ルール
1.  **プロジェクトルートの定義**: 本プロジェクトのルートディレクトリは `c:\tools\astah-plugins\stm` です。
2.  **絶対パスの強制**: ファイルの作成・更新（diff形式）を行う際は、必ず上記のプロジェクトルートを含む**完全な絶対パス**を指定してください。相対パスは使用しないでください。
3.  **パスの検証**: 出力前に、パスが `c:\tools\astah-plugins\stm\` で始まっているか、`c:\c\` のようにドライブ文字が重複していないか、`src/` が重複していないかを確認してください。
4.  **配置場所の厳守**:
    *   テストスクリプト (`.js`): `c:\tools\astah-plugins\stm\test\scripts\`
    *   テスト仕様書 (`.md`): `c:\tools\astah-plugins\stm\test\`
    *   単体テスト (`.java`): `c:\tools\astah-plugins\stm\src\test\java\` 配下の適切なパッケージディレクトリに配置する。
        *   Modelのテスト例: `c:\tools\astah-plugins\stm\src\test\java\snytng\astah\plugin\stm\model\`

## 11. 振り返り (Retrospective)
1.  **実施タイミング**: 1日の作業終了時、または大きな機能の実装完了時など、区切りの良いタイミングで実施する。
2.  **手法**: AI協働型KPT (Keep, Problem, Try) を用いる。
    *   **Keep (良かったこと・続けたいこと)**:
        *   AI視点: 助かった指示、正確に出力できた要因。
        *   人間視点: うまくいった指示パターン、効率的だった手順。
    *   **Problem (困ったこと・改善したいこと)**:
        *   AI視点: 誤解した点、エラーを出した点、非効率だった点。
        *   人間視点: 指示に迷った点、負担に感じた点。
    *   **Try (次回の挑戦・具体的なアクション)**:
        *   ルール変更: `development_rules.md` への追記・修正。
        *   プロンプト工夫: 次回試したい指示の出し方。
3.  **プロセス**: 私（人間）が振り返りを提案し、geminiさんがログを分析してKPT案を提示する。私がそれを確認・採用し、必要に応じてルールを更新する。
